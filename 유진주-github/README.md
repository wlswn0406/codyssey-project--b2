# 유진주

## 요약

| 항목 | 내용 |
|---|---|
| 주제 | AI |
| 키워드 | 보안, 규제, 모델 |
| 자동화 도구 | GitHub Actions |
| AI 모델 (요약) | Gemini API (gemini-1.5-flash) |
| AI 모델 (이미지) | Gemini Imagen (gemini-2.0-flash-preview-image-generation) |
| 뉴스 소스 | Google News, TechCrunch, VentureBeat (다중 RSS) |
| 데이터베이스 | GitHub Repository (마크다운 파일) + GitHub Pages 리포트 뷰어 |


### 도구 선택 이유

노코드 자동화 도구는 UI에서 블록을 연결하는 방식으로, 파이프라인이 내부적으로 어떻게 동작하는지 파악하기 어렵다.

노코드 도구를 학습 목적으로 사용하는 것은 개인적으로 의미가 없다고 판단했으며, 직접 코드를 작성하며 동작 원리를 이해하는 것이 더 재미있을 것 같아 선택했다.

GitHub Actions는 스케줄만 정의하고, RSS 파싱·API 호출·DB 저장 등 실제 처리 로직은 Python 스크립트로 작성한다. 시간 제약상 소스코드는 바이브코딩을 활용했다.


### 키워드 선택 이유

- **보안(취약점)**: AI 관련 보안 이슈를 통해 AI 기술의 리스크를 파악할 수 있다.
- **규제(정책)**: AI 규제 법안은 기술 개발 방향과 산업 전략에 실질적으로 영향을 미친다.
- **모델**: 새로운 AI 모델 출시와 업데이트는 기술 선택지에 직접 영향을 준다.



## 구현 과정

**흐름**
```text
[GitHub Actions] (스케줄러)
       │
       ▼
[Python 스크립트 (main.py)]
 1. 뉴스 수집: Google News, TechCrunch, VentureBeat RSS 다중 파싱
 2. 1차 필터링: 키워드 포함 여부 확인 + 중복 제거
 3. 중요도 평가: Gemini에 1~10점 요청 (filter_prompt.txt), 7점 미만 SKIP
 4. AI 요약: 통과된 기사를 3줄 요약 + 감성 태그 (summary_prompt.txt)
 5. DB 저장: 마크다운 파일 생성 후 리포지토리에 커밋 (Git Scraping)
 6. 발행: GitHub Pages로 자동 웹 페이지 발행
```

**구현 체크리스트**
- [x] 1. 환경 설정 (GitHub Secrets 등록, GitHub Action 생성)
- [ ] 2. 수집 구성 (다중 RSS 파싱 + 키워드 필터링 + 중복 제거)
- [ ] 3. 요약 연동 (filter_prompt 중요도 평가 → summary_prompt 3줄 요약)
- [ ] 4. 저장 연동 (마크다운 생성 + git commit & push)
- [ ] 5. 스케줄 설정 (GitHub Actions cron)
- [ ] 6. 보너스 (감성 분석 태그 + 썸네일 이미지 생성)
- [ ] 7. 발행 (GitHub Pages 세팅)


### 1. 환경 설정

**미션 요건**
- API 키/토큰은 환경 변수에 보관하며 문서에 노출 금지

**구현 계획**
- GitHub Actions 서버 시간대를 한국 시간(KST)으로 고정 (`TZ: Asia/Seoul`)
- GitHub Secrets에 `GEMINI_API_KEY` 등록 후, 워크플로우를 통해 파이썬 스크립트의 환경 변수로 안전하게 전달

### 2. 수집 구성

**미션 요건**
- 지정된 키워드 필터링 및 링크·제목·본문 포함
- 중복 방지 키 적용 (링크 해시)
- 기사 1건당 1회 호출 원칙

**구현 계획**
- 수집 로직을 실행하는 파이썬 스크립트 파일(`src/collect.py`)을 생성
- `feedparser` 라이브러리를 통해 Google News, TechCrunch, VentureBeat RSS를 순회
- 매일 오전 8시 수집 시간을 기준으로 최근 24시간 이내에 발행된 기사만 수집
- 기사 제목을 키워드 기반으로 필터링해 `logs/` 폴더에 `YYYYMMDD-수집.txt`에 제목, 링크, RSS 요약 기록
- `YYYYMMDD-수집.txt` 파일을 기준으로 동일한 링크의 기사는 중복 기록하지 않음
- 기사 수집 중 에러가 발생할 경우, 재시도 없이 건너뛰고(Skip) 다음 기사 수집을 이어가도록 예외 처리 적용
- 중요도 평가 로직을 실행하는 파이썬 스크립트 파일(`src/evaluate.py`)을 생성
- 순회가 끝나면 수집된 기사 정보(제목, 요약)와 `filter_prompt.txt`를 LLM에 전달해 중요도 평가(1~10점)를 요청해 2차 필터링
- 평가가 7점 이상인 기사만 선별하여 기사 본문과 추가 속성을 수집해 `logs/YYYYMMDD-필터링.txt`에 기록

**오류 처리 (에러 로깅 프로토콜)**
- 파이프라인 단계별 에러 추적을 위해, 스크립트 내부에 규약(Protocol)으로 오류 태그를 명시
- 기사 수집 실패 시: `logs/YYYYMMDD-수집.txt`에 `[수집 오류]` 태그와 사유 기록
- 중요도 평가/본문 수집 실패 시: (최대 2회 재시도 후) `logs/YYYYMMDD-필터링.txt`에 `[평가 오류]` 태그와 사유 기록
- 중요도 미달(7점 미만) 기사: 중복 평가를 방지하기 위해 `[STATUS] SKIP` 태그와 함께 기록하여 **1기사 1호출 원칙**을 엄격히 준수

### 3. 요약 연동 및 썸네일 생성

**미션 요건**
- 3줄 이내 요약 및 기사 맞춤형 썸네일 이미지 생성
- 기사 1건당 LLM 1회 호출 원칙 (프롬프트 최적화)

**구현 계획**
- 요약과 이미지 생성을 전담하는 파이썬 스크립트(`src/summarize.py`) 생성
- 중요도 7점을 통과한 기사 본문과 `summary_prompt.txt`를 LLM(Gemini)에 전달해 3줄 요약과 감성 분석 태그(#긍정/#부정/#중립) 반환
- `image_prompt.txt`를 이용해 기사 제목 기반의 프롬프트를 구성하고, 무료 이미지 생성 API를 통해 썸네일 이미지 URL 획득
- 두 결과를 취합하여 `logs/YYYYMMDD-요약.txt`에 `[SUMMARY]`와 `[IMAGE]` 태그로 저장
- `google-generativeai` 라이브러리를 사용하며, 프롬프트 파일은 파이썬 `open()` 함수로 읽어와 변수로 할당하여 사용

**오류 처리**
- 기사 특성상 단순 팩트 전달이거나 감성 판단이 애매한 경우, 억지로 긍정/부정으로 분류하지 않고 `#중립`으로 처리
- 요약 API 통신 에러 발생 시: (최대 2회 재시도 후) `logs/YYYYMMDD-요약.txt`에 `[요약 오류]` 태그와 사유 기록


### 4. 저장 연동

**미션 요건**
- 기사별로 제목, 3줄 요약, 감성 분석 태그, 원문 링크, 발행일시, 썸네일 이미지 속성 저장

**구현 계획**
- 마크다운 발행을 전담하는 파이썬 스크립트(`src/publish.py`) 생성
- `logs/YYYYMMDD-요약.txt`에 저장된 정상 기사 데이터만 파싱하여 마크다운 문서로 포맷팅
- `YYYY-MM-DD-news.md` 형태로 당일 발행된 다중 기사를 Append(이어쓰기) 방식으로 저장
- GitHub Actions를 통해 변경된 마크다운 파일과 로그 폴더를 자동 `git commit & push` 수행

**오류 처리**
- 에러 로그(`[요약 오류]` 등)나 무효한 데이터(`[STATUS] SKIP`)는 파싱 단계에서 안전하게 무시하도록 예외 처리


### 5. 파이프라인 자동화

**미션 요건**
- 사람의 개입 없이 지정된 시간에 전체 파이프라인이 자동 실행

**구현 계획**
- GitHub Actions 워크플로우(`.github/workflows/ai-news-pipeline.yml`) 구성
- `on: schedule` 옵션을 사용해 크론탭(cron) 문법으로 KST 기준 매일 오전 8시(UTC 23시)에 파이프라인 구동
- `src/collect.py` -> `src/evaluate.py` -> `src/summarize.py` -> `src/publish.py` 순으로 스크립트 순차 실행

**오류 처리**
- 각 Python 스크립트가 내부적으로 오류를 제어하고 정상 종료되도록 설계하여, 워크플로우 중간에 개별 기사 에러가 나더라도 전체 파이프라인이 중단되지 않고 끝까지 실행되도록 보장
